# Audania — Stack & Architecture Decisions

Distilled from the parent Audania monorepo's `CLAUDE.md` §4 and §9 decision log. The root is canonical — flag any drift.

## Framework and language

- **Laravel 13.** Pinned. Boost-detected ecosystem versions in the generated `CLAUDE.md` are authoritative for what is actually installed.
- **PHP 8.4.** Boost-detected default for Laravel 13.
- **Blade + Livewire** for the Praxis-facing UI. **No Inertia, no React, no Vue in this repo.** Boost ships skills for Inertia React / Vue, but they are deliberately not active here (`boost.json` confirms only `laravel-best-practices`, `pest-testing`, `tailwindcss-development`). Keep it that way — adding an SPA framework defeats the Livewire choice.
- **Vue 3** lives in the sibling repo `audania-patient-pwa`, not here. If you find yourself reaching for Vue or React in this app, stop — the surface you're working on is probably not in this app.

Rationale and trade-offs live in root `CLAUDE.md` §9 (2026-05-09 Laravel-13 / Livewire decision).

## Marketing pages live in this app

Marketing for `audania.de` lives **in this Laravel app** as a dedicated Blade route group, with its own layout (`layouts.marketing` or equivalent) and **no Livewire** (root `CLAUDE.md` §9, 2026-05-09 marketing-as-route-group decision).

Re-split triggers — situations that would justify extracting marketing into a separate service later — are documented in root §9: an SEO-driven static-site / CMS stack that wants its own deploy cadence; an attack-surface increase from anonymous traffic that route-group / Nginx isolation cannot contain; a non-engineering marketing function that needs to ship copy without going through the SaaS release pipeline. Until any of those fires, marketing is a route group here, not a service.

## Database

**MySQL 8.x**, EU-hosted, encrypted at rest, **row-level isolation per Praxis** (root `CLAUDE.md` §9, 2026-05-09 MySQL decision). Tenant-scoped tables include `praxis_id` as a NOT NULL FK with a composite index from the first migration — no retro-fitting.

Important consequence of the MySQL choice: there is no DB-layer Row-Level Security. Tenant isolation is **application-layer only**, enforced via the Eloquent global scope on `praxis_id` described in `audania-hard-rules.md`. The test discipline there ("scope filters cross-tenant rows"; "any `withoutGlobalScopes` usage is justified and audited") is the **only** enforcement gate — treat any failure or skip in those tests as a §2 hard-rule violation, not a regular test flake.

For Art. 9 health-data fields that need at-rest column encryption, use Laravel's `encrypted` cast (application-layer) rather than reaching for DB extensions. JSON columns (MySQL 8 native JSON type with virtual columns and indexes on JSON paths) are fine for structured payloads coming from `audania-dialog`.

## LLM inference is not in this repo

Dialog and slot-filling run in the sibling service `audania-dialog` (Python 3.12 + FastAPI). This app calls it over HTTP. Concrete consequences for this codebase:

- No `openai-php/*`, `anthropic-php/*`, or any LLM SDK in this `composer.json`.
- No direct LLM calls from a controller, command, or queue job in this app.
- The HTTP client to `audania-dialog` should be a single, narrow service class. Pseudonymise inputs at the call site — never pass a raw request body straight through.
- Slot definitions are not owned here. They live in the parent monorepo at `20-product/slot-definitions/` and are consumed by `audania-dialog` (root README "house rules").

## Hosting and data path

- **Hosting:** Hetzner Cloud (FRA / Nuremberg) by default. STACKIT, IONOS, OVH FRA acceptable second sources. Azure / AWS Frankfurt only with documented EU-residency contract.
- **Self-hosted assets:** Fonts come from `10-brand/visual-identity/fonts/` in the parent monorepo and ship bundled with the app. No Google Fonts CDN, no jsDelivr / unpkg CDNs, no Cloudflare in front of any path that touches Praxis or patient data.
- **Observability:** self-hosted Sentry or an EU-only SaaS with documented residency. Logs pseudonymised at write time (`audania-hard-rules.md`).

## Auth

- **Praxis staff:** OIDC.
- **Patient flows:** session-token only, no patient account. *But:* patient flows are not in this app — they live in `audania-patient-pwa`. If you're building a patient-auth flow here, double-check you're in the right repo first.

## Boost extension model — how this folder works

- The generated `CLAUDE.md`, `AGENTS.md`, `boost.json`, `.mcp.json`, and `junie/` are **git-ignored**. They are regenerated by `php artisan boost:install` / `boost:update`. Do not edit them directly — your edits will be overwritten.
- The canonical Audania-side context for AI agents lives in `.ai/guidelines/audania-*.md` (this folder). Boost merges these into the regenerated `CLAUDE.md` automatically (Laravel Boost docs, "Adding Custom AI Guidelines").
- After adding or editing a guideline file in `.ai/guidelines/`, run `php artisan boost:update` so the next Claude Code session picks the change up.
- Custom Audania-specific *skills* (on-demand patterns, e.g. for tenant-scoping or marketing-page scaffolding) belong in `.ai/skills/{skill-name}/SKILL.md` — but only once a recurring pattern has actually emerged. Do not create skills speculatively.
